# EduTrack QR ‚Äî QR-Based Attendance System

Django web app for classroom attendance via QR codes. Three roles: **Student**, **Teacher**, **Superuser (admin)**.

## Stack

- **Backend:** Django 6.0.5, Python
- **DB:** SQLite (dev) ‚Äî `psycopg2-binary` included, so Postgres-ready for prod
- **Frontend:** Django templates + Bootstrap 5.3.8 + Bootstrap Icons + custom CSS (`static/CSS/index.css`)
- **QR:** `qrcode` lib (`QRA_app/QR_generator.py`)
- **Tunneling:** `pyngrok` (likely for local demo/testing on real devices)
- **Auth:** Custom `User` model extending `AbstractUser` with `role` field (student/teacher/superuser)

## Data Model (core relations)

- `User` (role: student/teacher/superuser) ‚Üí auto-sets role from `is_staff`/`is_superuser`
- `DepartmentModel` ‚Üí has many `CourseModel`
- `CourseModel` ‚Üí belongs to a department, has `CourseAssignment` (teacher) and `EnrollmentModel` (students)
- `AttendanceSessionModel` ‚Üí one per class session, has a unique `session_token` (UUID) used to build the QR/scan URL, with `expires_at` + `is_active`
- `AttendanceRecordModel` ‚Üí one row per student per session (`present`/`absent`), unique per (student, session)
- `DeviceLogModel` ‚Üí tracks device hash per user (anti-proxy/spoofing check, presumably)
- `AttendanceSummaryModel` ‚Üí cached `attendance_percentage` per (student, course)

**Flow:** Teacher starts a session ‚Üí QR generated from `session_token` ‚Üí student scans ‚Üí hits `/student/scan/<token>` ‚Üí attendance record created if session valid/not expired.

## Recent Feature Additions

### 1. Desktop ‚Üî Mobile QR Handoff (student dashboard)
Scanning a QR code with a laptop webcam is a bad experience, so the student dashboard now detects device type server-side (`is_mobile()` in `students_views.py`, user-agent check) and branches:
- **Desktop:** shows the *current* active session's QR code (generated in-memory as a base64 PNG via `generate_qr_base64()` in `QR_generator.py` ‚Äî no file saved to disk), with a "Waiting for scan" status that polls `session_status` every 3s and flips to "Checked in" once the phone scan lands. Student logs into the same dashboard on their phone and scans the code shown on their own laptop screen.
- **Mobile:** unchanged ‚Äî shows the existing camera scanner (`html5-qrcode`).
- New endpoint: `student/session/<id>/status/` (`session_status` view) ‚Äî lightweight JSON polling, no page reload needed.
- **Known limitation:** desktop QR only appears for courses the student is already enrolled in. See roster feature below ‚Äî this is now resolved as long as a teacher pre-enrolls the student.

### 2. Roster-Based Enrollment (security fix)
Previously, `submit_attendance` auto-enrolled *any* student in *any* course the instant they scanned a valid token (`EnrollmentModel.objects.get_or_create(...)`). That meant a leaked/screenshotted/shared QR code let anyone mark themselves present in a class they were never part of.

Fixed by flipping to roster-first:
- `submit_attendance` now checks `EnrollmentModel.objects.filter(student=student, course=session.course).exists()` and rejects the scan with a flash message if the student isn't already enrolled ‚Äî no more silent self-enrollment.
- Teachers manage their own course rosters at `teacher/course/<course_id>/roster/` (new `manage_roster` / `enroll_student` / `remove_student` views + `manage_roster.html` template). Lookups are scoped to `courseassignment__teacher=request.user`, so a teacher can only manage rosters for courses they're actually assigned to.
- "Roster" button added to the course card on `teacher_dashboard.html`, next to Start/Details, matching existing card style.
- Django admin (`/admin/`) also works as a fallback roster tool since `EnrollmentModel` is already registered there.
